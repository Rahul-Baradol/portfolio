Have you ever noticed how effortlessly apps like WhatsApp let you scroll through thousands of messages? 

We often don't think about the implementation behind these things, but there is a bit of complexity involved.

To understand that, let's build a simple message viewer and see why rendering large lists is harder than it looks.

## Starting by defining some constants...

```jsx
export enum MessageCountOption {
    "TEN" = 10,
    "HUNDRED" = 100,
    "THOUSAND" = 1000,
    "TEN_THOUSAND" = 10000
}

export const dummyMessages = [
    "Hello, how are you?",
    "I'm fine, thank you!",
    "What are you up to today?",
    "Just working on some projects.",
    "That sounds interesting!"
]

export function generateMessages(length: MessageCountOption): Message[] {
    return  Array.from({ length: length }, (_, index) => ({
        agent: index % 2 === 0 ? "me" : "friend",
        content: `${index + 1}. ${dummyMessages[index % dummyMessages.length]}`
    }))
}
```

## So the first step of optimization is this...

Can we make sure that only the messages which are visible in the viewport are rendered in the DOM?

My first idea was simple: render only a fixed number of messages, the ones that actually fit inside the viewport.

Let's say 8 for this container, because they perfectly fit within the visible area.

We attach a wheel event listener to the container, which is responsible for incrementing/decrementing a state called **scrollPosition**

```jsx
const [scrollPosition, setScrollPosition] = useState<number>(0);
```

```jsx
useEffect(() => {
    const container = containerRef.current;
    if (!container) {
        return;
    }

    const handleWheel = (e: WheelEvent) => {
        if (!container) {
            return;
        }

        if (e.deltaY > 0) {
            setScrollPosition(ele => ele + 1);
        } else {
            setScrollPosition(ele => Math.max(ele - 1, 0));
        }
    };

    const handlePointerDown = (e: PointerEvent) => {
        startYRef.current = e.clientY;
        container.setPointerCapture(e.pointerId);
    };

    const handlePointerUp = (e: PointerEvent) => {
        if (startYRef.current === null) {
            return;
        }

        const deltaY = startYRef.current - e.clientY;

        container.releasePointerCapture(e.pointerId);
        startYRef.current = null;

        if (deltaY > 0) {
            setScrollPosition((p) => p + 10);
        } else {
            setScrollPosition((p) => Math.max(p - 10, 0));
        }
    };

    const handlePointerCancel = (e: PointerEvent) => {
        container.releasePointerCapture(e.pointerId);
        startYRef.current = null;
    };

    containerRef.current?.addEventListener("wheel", handleWheel);
    container.addEventListener("pointerdown", handlePointerDown);
    container.addEventListener("pointerup", handlePointerUp);
    container.addEventListener("pointercancel", handlePointerCancel);
    return () => {
        containerRef.current?.removeEventListener("wheel", handleWheel);
        container.removeEventListener("pointerdown", handlePointerDown);
        container.removeEventListener("pointerup", handlePointerUp);
        container.removeEventListener("pointercancel", handlePointerCancel);
    };
}, []);
```

You can see that we are also attaching pointer event listeners to the container.

To make sure it works for different inputs like "touch" on mobile phones.

We keep track of the Y position of the pointer when the user starts scrolling, and then when the user stops scrolling, we calculate the difference to determine how much to scroll, and update the scrollPosition.

You can see that we are mutating the state **visibleMessages** just like before, and doing a sliding window of what messages to show.

But how is this different from previous approach?

Well the messages won't snap, like before, because their position is with respect to the position of the parent element in the DOM, which is outer div. Unlike previous approach, where one message's position was determined by the message above it, causing the snapping.

If you observe, then the transform value for a given message would be the same, and would not change.

## Some tradeoffs worth mentioning. 

When you seek into a far away message, you can see the div remains empty for a split second, and then we see messages.

This is something we see with **react-window** as well, and it is a tradeoff. We can decide how many elements to render in advance (even the ones which are out of viewport), to prevent the "empty container" state in exchange for memory consumed. 

The number of messages here is the **overscan**, and react-window allows the user to tweak it.

Higher overscan, means we render more items in advance, which can help with smooth scrolling, but also increases memory usage.

Lower overscan, means we render fewer items in advance, which can reduce memory usage but may lead to less smooth scrolling.

### We made one important assumption: 
Every row has a fixed height, which lets us calculate the total scrollable height ahead of time.

In the real world, messages often have different heights. Supporting that efficiently is one of the reasons libraries like react-window and react-virtualized become considerably more sophisticated.

Even so, this experiment conveys the core idea behind list virtualization and helps develop a mental model for understanding how virtualization works.
