*This is inspired from an experiment on Codrops. Link in resources!*

The experiment is simple, being able to create a gravity-based mouse trail, where images are spawned following the mouse cursor.

Although they have used GSAP with no React environment, I thought let's give it a try in React, starting off with naive methods, and gradually moving to optimized methods.

## Starting off with naive implementation

What all things we need?
- How to keep track of images in action ?
- How to animate them ?

Every image shares the same core shape, the bits the physics needs to move it around. Let's call that the `PhysicsImage`.

```jsx
interface PhysicsImage {
    x: number; // current x position
    y: number; // current y position
    dx: number; // change in x position
    dy: number; // change in y position
    src: string; // image source
    stage: 'initial-bounce-up' | 'free-fall' | 'bounce-up' | 'bounce-down'; // stage of the animation
}
```

Each approach we build will need a few extra fields on top of this ( like an `id`, or DOM bookkeeping ), so they'll just extend the base. For the naive approach, that looks like

```jsx
interface NaiveImage extends PhysicsImage {
    id: number; // unique identifier, used as the React key
    imageId: number; // id of the source image
}
```

We'll see why later approaches tweak these extra fields, but the `PhysicsImage` base stays the same throughout, and that's exactly what lets us share the physics logic across every approach.

And since we would be simulating some physics ( like gravity ), some constants would help 
```jsx
export const HORIZONTAL_SPREAD = 5;

export const INITIAL_FALL_SPEED = 5;
export const INITIAL_BOUNCE_UP_SPEED = 10;

export const GRAVITY = 1;         
export const HORIZONTAL_FRICTION = 0.98;
export const SPAWN_THROTTLE_MS = 100;   
export const IMAGE_SIZE_PIXELS = 100;   
```    

Maintain a state of these **images**

```jsx
const [images, setImages] = useState<NaiveImage[]>([]);
```

Some references to keep track of last spawn time, container element, a counter to hand out unique ids, and the current animation frame
```jsx
const lastSpawnTime = useRef<number>(0);
const nextId = useRef<number>(0);
const animationFrameIdRef = useRef<number | null>(null);
```

Each spawned image gets a fresh `id` from `nextId`, and the chosen source image's id is kept separately as `imageId`. This is because we use `id` as the React `key`, and since many spawned images can share the same source image, the source id alone wouldn't be unique.

And then for each mouse event, we would "spawn" images by populating `images` state

```jsx
const handleMouseMove = (event: React.MouseEvent) => {
    const container = containerRef.current;
    if (!container) {
        return;
    }

    const rect = container.getBoundingClientRect();
    const curX = event.clientX - rect.left;
    const curY = event.clientY - rect.top;

    const now = performance.now();
    if (now - lastSpawnTime.current < SPAWN_THROTTLE_MS) {
        return;
    }

    lastSpawnTime.current = now;

    const randomImageIndex = Math.floor(Math.random() * imageLinks.length); 

    const newImage: NaiveImage = {
        x: curX - (IMAGE_SIZE_PIXELS / 2),
        y: curY - (IMAGE_SIZE_PIXELS / 2),
        imageId: imageLinks[randomImageIndex].id,
        id: nextId.current++,
        src: imageLinks[randomImageIndex].src,
        stage: 'initial-bounce-up',
        dx: (Math.random() * HORIZONTAL_SPREAD * 2) - HORIZONTAL_SPREAD,
        dy: INITIAL_BOUNCE_UP_SPEED
    };

    setImages(prevImages => {
        return [...prevImages, newImage];
    });
}
```

Think of imageLinks as an array of image objects, each with an `id` and `src` property.
```jsx
export const imageLinks = [
    {
        id: 1,
        src: '/experiments/mouse-trail/01.png'

    ...
];
```

Now, the actual physics, advancing an image by one frame, is identical no matter how we end up rendering it. So instead of inlining it into every render loop, let's pull it into one pure function that mutates a `PhysicsImage` in place.

```jsx
export function stepImage<T extends PhysicsImage>(image: T, floor: number) {
    switch (image.stage) {
        case 'initial-bounce-up':
            image.y -= image.dy;
            image.x += image.dx;

            image.dy -= GRAVITY;
            image.dx *= HORIZONTAL_FRICTION;
            if (image.dy <= 0) {
                image.stage = 'free-fall';
                image.dy = INITIAL_FALL_SPEED;
            }
            break;

        case 'free-fall':
            image.y += image.dy;
            image.dy += GRAVITY;

            if (image.y >= (floor - 100)) {
                image.stage = 'bounce-up';
                image.dy = INITIAL_BOUNCE_UP_SPEED;
            }
            break;
        
        case 'bounce-up':
            image.y -= image.dy;
            image.dy -= GRAVITY;
            if (image.dy <= 0) {
                image.stage = 'bounce-down';
                image.dy = INITIAL_FALL_SPEED;
            }
            break;

        case 'bounce-down':
            image.y += image.dy;
            image.dy += GRAVITY;
    }
}
```

Because it accepts any `PhysicsImage`, every variant we build later, with its own extra fields, can be stepped by this exact same function. From here on, whenever you see `stepImage(image, floor)` in a render loop, this is what it's doing.

We start the render loop like this

```jsx
useEffect(() => {
    const container = containerRef.current;
    if (!container) {
        return;
    }

    const renderLoop = () => {
        const floor = container.clientHeight ?? 0;

        setImages(prevImages => {
            if (prevImages.length === 0) {
                return [];
            }

            let updatedImages = prevImages.map(image => {
                const newImage = {
                    ...image,
                }

                stepImage(newImage, floor);

                return newImage;
            });

            updatedImages = updatedImages.filter(image => image.y < floor);

            return updatedImages;
        });

        animationFrameIdRef.current = requestAnimationFrame(renderLoop);
    }

    const startOrStopAnimation = () => {
        const shouldAnimate = isVisibleRef.current && 
                                isIntersectingRef.current &&
                                !prefersReduced;

        if (shouldAnimate && !animationFrameIdRef.current) {
            console.log("Starting RAF");
            animationFrameIdRef.current = requestAnimationFrame(renderLoop);
        }

        if (!shouldAnimate && animationFrameIdRef.current) {
            console.log("Stopping RAF");
            cancelAnimationFrame(animationFrameIdRef.current);
            animationFrameIdRef.current = null;
        }
    };

    const observer = new IntersectionObserver(([entry]) => {
        isIntersectingRef.current = entry.isIntersecting;
        startOrStopAnimation();
    });

    observer.observe(container);

    const handleVisibilityChange = () => {
        isVisibleRef.current = document.visibilityState === "visible";
        startOrStopAnimation();
    };

    document.addEventListener("visibilitychange", handleVisibilityChange);

    return () => {
        if (animationFrameIdRef.current) {
            cancelAnimationFrame(animationFrameIdRef.current);
        }
        observer.disconnect();
        document.removeEventListener("visibilitychange", handleVisibilityChange);
    }
}, []);
```

Notice we don't start `requestAnimationFrame` unconditionally. Instead we gate it on visibility with an `IntersectionObserver`, so the loop only runs while the demo is actually on screen, and gets cancelled the moment it scrolls out of view.

We also listen for the `visibilitychange` event on the document to pause the animation when the tab is not visible.

This matters because this page stacks several of these demos on top of each other. Without gating, every one of them would be running its render loop at once, even the ones you can't see, all fighting over the same frame budget. We apply this same visibility-gating to every approach that follows, so I'll leave it out of the snippets from here on to keep them focused.

And we render these as follows

Now before we get into optimizations, let's take a look at the current implementation and identify the bottlenecks.

1. **Frequent Layout Calculations**: The current implementation triggers layout calculations on every frame, which is expensive. This is especially true for the "Layout" stage, which can be a significant performance bottleneck.

2. **Image Management**: The way images are managed and updated in the state can lead to unnecessary re-renders, increasing scripting time (because of react's reconciliation) and high rendering time. Each image's position is updated individually, which can be optimized. 

3. **Rendering Pipeline**: The naive approach goes through the entire rendering pipeline (DOM > Style Calculation > Layout > Paint > Composite) for each frame, leading to potential performance issues.

## Bypassing React with direct DOM updates

So the next phase would be where we optimize the way we store and manage the images in the state.
Let's not use React "state" at all, and modify the DOM directly, bypassing React's reconciliation.

Why?

Because React's reconciliation process can introduce latency, especially with frequent updates like animations.
Here we are trying to achieve 60fps, which means we have around 16ms for a frame to update the positions of all images and render them. React's reconciliation can easily exceed this time budget when there are many images, leading to dropped frames and a janky animation.

So how to do update DOM directly?

First we maintain the images as a React ref, which doesn't trigger a React re-render on update

```jsx
const images = useRef<UncontrolledImage[]>([]);
```

We also keep a incrementing counter for assigning unique ids to images
```jsx
const nextDomId = useRef<number>(0);
```

The extra fields shift a bit here too, but the `PhysicsImage` base stays untouched. Since we are no longer relying on React's reconciliation, we don't need the React `key`, so the `id` field is dropped. Instead we add a `domId` (from `nextDomId`) to map each image to its DOM node, along with two bookkeeping flags, `insertedInDom` and `shouldBeDeleted`, to track its lifecycle in the DOM.

```jsx
interface UncontrolledImage extends PhysicsImage {
    domId: number; // maps the image to its DOM node
    insertedInDom: boolean; // has it been appended to the DOM yet?
    shouldBeDeleted: boolean; // is it ready to be removed?
}
```

And for each mouse move, we do the following where we update the ref and doesn't trigger a re-render 
```jsx
const handleMouseMove = (event: React.MouseEvent) => {
    if (!paintElement.current) {
        return;
    }
    const rect = paintElement.current.getBoundingClientRect();
    const curX = event.clientX - rect.left;
    const curY = event.clientY - rect.top;

    const now = performance.now();
    if (now - lastSpawnTime.current < SPAWN_THROTTLE_MS) {
        return;
    }

    lastSpawnTime.current = now;

    const randomImageIndex = Math.floor(Math.random() * imageLinks.length); 

    const newImage: UncontrolledImage = {
        x: curX - (IMAGE_SIZE_PIXELS / 2),
        y: curY - (IMAGE_SIZE_PIXELS / 2),
        domId: nextDomId.current++,
        src: imageLinks[randomImageIndex].src,
        stage: 'initial-bounce-up',
        dx: (Math.random() * HORIZONTAL_SPREAD * 2) - HORIZONTAL_SPREAD,
        dy: INITIAL_BOUNCE_UP_SPEED,
        insertedInDom: false,
        shouldBeDeleted: false,
    };

    images.current.push(newImage);
}
```

And then in the render loop, we iterate through the images in ref every frame, and update DOM directly.
```jsx
useEffect(() => {
    if (paintElement.current) {
        const renderLoop = () => {
            const floor = paintElement.current!.clientHeight;

            images.current.forEach((image) => {    
                stepImage(image, floor);

                if (image.insertedInDom === false) {
                    const domImage = document.createElement('img');
                    domImage.id = `image-${image.domId}`;
                    domImage.src = image.src;
                    domImage.style.position = 'absolute';
                    domImage.style.width = `${IMAGE_SIZE_PIXELS}px`;
                    domImage.style.height = `${IMAGE_SIZE_PIXELS}px`;
                    domImage.style.left = `0px`;
                    domImage.style.top = `0px`;
                    domImage.style.transform = `translate(${image.x}px, ${image.y}px)`;
                    paintElement.current!.appendChild(domImage);

                    imageCache.current.set(image.domId, domImage);

                    image.insertedInDom = true;
                } else if (image.y < floor) {
                    const domImage = imageCache.current.get(image.domId);
                    if (domImage) {
                        domImage.style.transform = `translate(${image.x}px, ${image.y}px)`;
                    }
                } else {
                    const domImage = imageCache.current.get(image.domId);
                    if (domImage) {
                        paintElement.current!.removeChild(domImage);
                        imageCache.current.delete(image.domId);
                        image.shouldBeDeleted = true;
                    }
                }
            });

            images.current = images.current.filter(image => !image.shouldBeDeleted);

            animationFrameIdRef.current = requestAnimationFrame(renderLoop);
        }

        // loop start + cleanup: same IntersectionObserver visibility-gating as before
    }
}, []);
```

You might have noticed, we are selecting the DOM image via in-memory Map and not via document object's query selector.
That is to avoid *O(N)* DOM scan, and directly access the DOM element we want to update, because remember we have to do all these operations under 16.6ms to achieve 60fps!

## Moving to canvas

The reason is because, in canvas approach we would handle the rendering ourselves, giving us more control over what operation to perform per frame.

That also means, we have to be careful in memory handling, making sure we don't create GC pressure.

In canvas approach, we would still store images as a React ref, handle the mouse event the same way as before, but the way we handle rendering would change significantly. Instead of relying on the DOM, we would draw the images directly onto the canvas.

The render loops changes to the following
```jsx
const renderLoop = () => {
    const floor = containerRef.current!.clientHeight;

    images.current.forEach(image => {
        stepImage(image, floor);
    });

    images.current = images.current.filter(image => image.y < floor);

    const canvas = canvasRef.current;
    const ctx = canvasContextRef.current;
    if (!canvas || !ctx) {
        return;
    }

    ctx.clearRect(0, 0, canvas.width, canvas.height);

    images.current.forEach(image => {
        const img = imageCache.current.get(image.src);
        if (img) {
            ctx.drawImage(img, image.x, image.y, IMAGE_SIZE_PIXELS, IMAGE_SIZE_PIXELS);
        }
    });

    animationFrameRef.current = requestAnimationFrame(renderLoop);
}
```

A couple of things to note here. We grab the canvas's 2D context once and put it in `canvasContextRef`, instead of calling `canvas.getContext("2d")` on every frame; cheap, but there's no reason to pay it 60+ times a second. 

And we have separated the *renderLoop* from the useEffect
```jsx
useEffect(() => {

    if (!canvas || !containerRef.current) {
        return;
    }

    canvasContextRef.current = canvas.getContext("2d");

    imageLinks.forEach(link => {
        if (!imageCache.current.has(link.src)) {
            const img = new Image();
            img.src = link.src;
            imageCache.current.set(link.src, img);
        }
    });

    const handleWindowResize = () => {
        const canvas = canvasRef.current;
        const container = containerRef.current;

        if (!canvas || !container) {
            return;
        }

        const dpr = window.devicePixelRatio || 1;

        const width = container.clientWidth;
        const height = container.clientHeight;

        canvas.style.width = `${width}px`;
        canvas.style.height = `${height}px`;

        canvas.width = width * dpr;
        canvas.height = height * dpr;

        const ctx = canvas.getContext("2d");
        if (!ctx) {
            console.error("Failed to get canvas context");
            return;
        }

        ctx.setTransform(1, 0, 0, 1, 0, 0);
        ctx.scale(dpr, dpr);
    }

    handleWindowResize();

    window.addEventListener("resize", handleWindowResize);

    // loop start + cleanup: same IntersectionObserver visibility-gating as before

    return () => {
        imageCache.current.clear(); // Also making sure we clear the image cache while unmounting
        window.removeEventListener("resize", handleWindowResize);
    }
}, []);
```

newImage.y += newImage.dy;

would run 60 times on a Mac and 120 times on a Dell laptop, in a second...

We need to fix this!

## Making it refresh-rate independent

We start with the following line, where we keep track of "when was last frame started"
```jsx
const lastFrameTime = useRef<number | null>(null);
```

We do this to calculate "how much time has elapsed since the last frame"

With that, we can estimate frame time! And in 60fps, the frame time is around 16.67ms.

So if frame time is 16.67ms, we can say frame rate is around 60fps.

If frame time is 8.33ms, we can say frame rate is around 120fps.

If frame time is 33.33ms, we can say frame rate is around 30fps.

And we want to scale the physics based on the frame time, so that the images fall at the same speed, regardless of the frame rate.
What frame rate we want to enforce? Let's do it 60fps.

The current logic works well on 60fps.
For example, look at the following line
```jsx
newImage.y += newImage.dy;
```

This works in 60fps. But breaks on 120fps, because it runs twice as much.
So what can we do? Update by **newImage.dy / 2** 

What if we run it on 30fps? Then we need to update by **newImage.dy * 2**

So this multiplier, can be derived as follows...
```jsx
(now - lastFrameTime.current) / FRAME_DURATION
```

Where `now` is the current frame time, `lastFrameTime.current` is the last frame time, and `FRAME_DURATION` is the duration of a frame at 60fps (16.67ms).

With this we are normalizing the position updates, to work on any frame rate, and we can call this multiplier as `frameFactor`

Since the physics now needs to scale every step by this `frameFactor`, we make a frame-rate-aware sibling of `stepImage`. Same structure, same stages, just every delta multiplied by `frameFactor` ( and the friction raised to its power, since it's multiplicative ).

```jsx
export function stepImageWithFrameFactor<T extends PhysicsImage>(image: T, floor: number, frameFactor: number) {
    switch (image.stage) {
        case 'initial-bounce-up':
            image.y -= image.dy * frameFactor;
            image.x += image.dx * frameFactor;

            image.dx *= Math.pow(HORIZONTAL_FRICTION, frameFactor);
            image.dy -= GRAVITY * frameFactor;

            if (image.dy <= 0) {
                image.stage = 'free-fall';
                image.dy = INITIAL_FALL_SPEED;
            }
            break;

        case 'free-fall':
            image.y += image.dy * frameFactor;
            image.dy += GRAVITY * frameFactor;

            if (image.y >= (floor - 100)) {
                image.stage = 'bounce-up';
                image.dy = INITIAL_BOUNCE_UP_SPEED;
            }
            break;

        case 'bounce-up':
            image.y -= image.dy * frameFactor;
            image.dy -= GRAVITY * frameFactor;

            if (image.dy <= 0) {
                image.stage = 'bounce-down';
                image.dy = INITIAL_FALL_SPEED;
            }
            break;

        case 'bounce-down':
            image.y += image.dy * frameFactor;
            image.dy += GRAVITY * frameFactor;
            break;
    }
}
```

And the render loop would be 
```jsx
const renderLoop = () => {
    const now = performance.now();
    let frameFactor = lastFrameTime.current ? (now - lastFrameTime.current) / FRAME_DURATION : 1;
    frameFactor = Math.min(frameFactor, 2);
    lastFrameTime.current = now;

    const floor = containerRef.current!.clientHeight;

    images.current.forEach(image => {
        stepImageWithFrameFactor(image, floor, frameFactor);
    });

    images.current = images.current.filter(image => image.y < floor);

    const canvas = canvasRef.current;
    const ctx = canvasContextRef.current;
    if (!canvas || !ctx) {
        return;
    }

    ctx.clearRect(0, 0, canvas.width, canvas.height);

    images.current.forEach(image => {
        const img = imageCache.current.get(image.src);
        if (img) {
            ctx.drawImage(img, image.x, image.y, IMAGE_SIZE_PIXELS, IMAGE_SIZE_PIXELS);
        }
    });

    animationFrameRef.current = requestAnimationFrame(renderLoop);
};
```

where we enforce 60Hz by adding this `frameFactor` 

Also we do 
```jsx
frameFactor = Math.min(frameFactor, 2);
```
to prevent overshooting of the frameFactor.

Imagine the user switches tab, does something else for a 1 minute, and comes back.

Then the `lastFrameTime.current` will be 1 minute behind the `now` time, leading to a very high `frameFactor`. 

So we need to clamp the `frameFactor` to a reasonable maximum value, to prevent any sudden jumps in the animation.
